# 第 12 章　从骨架起手新项目

前面十一章讲的是**方式 B**——从空目录 `composer create-project` 一步步搭。本章讲**方式 A 的正式版**：
把骨架当「标准起手框架」，走 **clone → 改名 → 配 env → 直接跑** 这条最短路径，
几分钟起一个干净的新项目，不再从零搭一遍。

> **这章跟第 1 章有什么不一样？**
> 第 1 章是「教你怎么搭出这套骨架」；本章是「骨架已经搭好了，教你怎么把它变成**你自己的新项目**」。
> 两条路殊途同归，按需选一条：想理解每块怎么来的走方式 B（第 1~11 章）；想立刻开工走本章。

> 本章同样遵守全教程的两条铁律：**每一步都有命令与结果**、**每一步都真机验证**。
> 下文所有命令与输出都在一个假想新项目 `acme/demo-shop` 上真实跑过（用 sqlite 起库，`57 passed` / `check 5/5` / 登录 `HTTP 200`），
> 卡住的地方就地记成「坑」，坑号对应目录页 [`README.md`](./README.md) 的「踩过的坑速查」。

## 12.1 起手总清单（先看全景）

一次起手 = 六组动作，按顺序过一遍即可。每组都是可勾选的 checklist，后面各节逐组展开：

```
□ 改名五件套    composer name / namespace 核查 / APP_NAME / README / CLAUDE.md 项目定位
□ 密钥三件套    APP_KEY / JWT_SECRET / 雪花三参（SNOW_FLAKE_*）
□ 双 composer   核对/定制 composer.production.json（生产源指向你自己的私包配置）
□ 外部接线      moo-monitor token 置空/占位 / gitee 远端 / git init fresh
□ 装依赖建库    composer install / migrate --seed（sqlite 或临时库）
□ 首次冒烟五命令 moo-system check / pest / migrate:status / route:list / 登录 200
□ 样例处置      教学样例 Food 二选一：保留作参考 / 按 12.9 清单删除
□ 建档          notes.md 起一条起手记录
```

第一步先把骨架弄到本地、剥掉它的 git 历史（你要的是代码起点，不是骨架的提交史）：

```bash
git clone git@gitee.com:charsen/moo-engine-skeleton.git demo-shop
cd demo-shop
rm -rf .git            # 丢掉骨架历史；12.6 会 git init 重开
```

> 之后所有 `composer` / `php artisan` 命令都在 **`engine/` 子目录**里执行（骨架的目录约定，见目录页）。

## 12.2 改名五件套

> **为什么单列一节讲改名**：宿主项目实战中曾出现过——一个项目的 `composer.json` `name` 沿用了模板的包名**很多年没改**，
> 直到某次排查依赖才发现「这项目对外的身份一直是模板」。改名清单就是为杜绝此类问题：起手第一件事就把身份改成自己的，一次到位。

**① `composer.json` 的 `name`**（两份都要改，见 12.5）：

```jsonc
// engine/composer.json 与 engine/composer.production.json 顶部
"name": "acme/demo-shop",                       // laravel/laravel → 你的 <vendor>/<project>
"description": "demo-shop backend.",            // 顺手改掉模板描述
```

> ⚠️ **坑 #30**：改完 `name` 后 `composer validate` 会报 **`composer.lock is not up to date`**——
> 因为 Composer 的 lock 内容哈希把 `name` 也算进去了（不只是 `require`）。这不是依赖变了，是哈希对不上。
> 解法：`composer update --lock`（只刷新 lock 的哈希、不升级任何包版本），再 `composer validate` 即 `valid`。
> 另外：`name` 建议用编辑器**手改那一行**，别用「读 JSON → 改 → 写回」的脚本——`json_encode` 会把整份文件重排版（数组展开、缩进变化），
> 徒增无谓 diff。

**② namespace 核查**：本骨架的应用命名空间是 Laravel 默认的 **`App\`**（见 `composer.json` 的 `autoload.psr-4`），
改项目名**不需要**改命名空间——`App\` 保持不动即可。核查一下确认没人把它改成别的：

```bash
grep '"App' engine/composer.json          # 期望仍是 "App\\": "app/"
```

**③ `APP_NAME`**（`engine/.env` 与 `engine/.env.example` 各一处）：

```dotenv
APP_NAME=demo-shop
```

**④ `engine/README.md`**：骨架自带一份最小 README，改成你项目的一句话说明（别把「骨架教程」的措辞带进新项目）。

**⑤ 根 `CLAUDE.md` / `README.md` 的项目定位**：如果你用 AI 协作，根 `CLAUDE.md` 顶部「这是什么」段要改成**你的**项目背景与路线，
否则 AI 会一直以为自己在「骨架教程仓」里干活。根 `README.md` 同理。

改名五件套勾选：

```
□ composer.json name + description（两份文件）
□ namespace 核查（App\ 不动）
□ APP_NAME（.env + .env.example）
□ engine/README.md 一句话说明
□ 根 CLAUDE.md / README.md 项目定位
```

## 12.3 密钥三件套

新项目**绝不能**沿用骨架的 `APP_KEY` / `JWT_SECRET`（那等于把门锁的钥匙公开挂在模板里）。三样都要重新生成/设定：

```bash
cd engine
cp .env.example .env        # 若还没有 .env
php artisan key:generate    # ① 写入全新 APP_KEY
php artisan jwt:secret --force   # ② 写入全新 JWT_SECRET（--force 覆盖 .env 里的空值/旧值）
```

**③ 雪花三参**（`SNOW_FLAKE_*`）——主键 ID 的生成参数，`.env` 里补三行（`config/snowflake.php` 有默认值，但生产必须显式设定）：

```dotenv
SNOW_FLAKE_DATA_CENTER_ID=1      # data_center_id：0-31
SNOW_FLAKE_WORKER_ID=1           # worker_id：0-31
SNOW_FLAKE_START_TIME=2026-01-01 # 纪元基准，新项目自定一个「不晚于今天」的日期
```

> ⚠️ 雪花两条「血泪」（`config/snowflake.php` 注释里也钉着，务必读一遍）：
> - **`start_time` 一旦线上生成过 ID 就永不可改**——改了会让新 ID 时间戳整体偏移、可能与历史 ID 撞主键。起手时定好、以后别动。
> - **多机部署时每台的 `(DATA_CENTER_ID, WORKER_ID)` 组合必须唯一**（各 5 bit），否则同毫秒会撞 ID。单机开发 `1/1` 即可，上多台前给每台不同的 `WORKER_ID`。

密钥三件套勾选：

```
□ APP_KEY（key:generate）
□ JWT_SECRET（jwt:secret --force）
□ 雪花三参（SNOW_FLAKE_DATA_CENTER_ID / WORKER_ID / START_TIME）
```

> 验证雪花生效：12.8 登录成功后，返回的 `data.user.id` 是一串十几位的雪花数（如 `72146523823149056`），
> 不是 `1`、`2` 这种自增值——说明主键走的是你新设的纪元。

## 12.4 双 composer 文件：本地开发 ↔ 生产部署

骨架带**两份** composer 文件，分工明确（完整机制见根 [`PRIVATE-COMPOSER-PACKAGES.md`](../PRIVATE-COMPOSER-PACKAGES.md) §2、§3，这里只讲起手要做什么）：

| 文件 | 谁用 | moo-* 版本约束 | `repositories` |
| --- | --- | --- | --- |
| `engine/composer.json` | **本地开发**（默认） | `dev-master as *`（追 master；本地想改包源码可改成 `path`） | vcs / path |
| `engine/composer.production.json` | **生产部署** | caret 锁稳定版（如 `^2.1.3`、可复现） | vcs |

**切换机制**：部署脚本 `pull.sh` 会读 `.env` 的 `APP_ENV`（或 `--production` 显式声明）判定生产，
在 Step 4 把 `composer.production.json` 的内容 `cp` 覆盖到 `composer.json`，再 `composer install`。
所以**生产机上 `git status` 长期显示 `M engine/composer.json` 是预期现象、不是 bug**（`pull.sh` 已自动过滤这一条，
且生产侧**绝不要** commit 这个改动，详见 PRIVATE-COMPOSER-PACKAGES.md §3.1）。

**起手要做的一件事——核对/定制 `composer.production.json`**：

- 把 `name` 改成你的（12.2 已提，两份都改）；
- **把生产的私包源指向你自己的配置**：`repositories.<key>.url` 与 `require` 的版本 pin 换成你有权限、且已验证可复现的版本。
  骨架当前的生产 pin 为 `moo-scaffold ^2.1.3` / `moo-monitor-laravel ^0.1` / `moo-system ^1.6.16`（本任务已把 moo-system
  从 `^1.6.15` 校准到 `^1.6.16`，对齐会话撤销黑名单回归基线）——你的项目按自己依赖的实际稳定版重新评估。

**两份都要能 `validate`**（起手时就跑一遍，别等部署才发现漂移）：

```bash
composer validate --no-check-publish composer.json              # composer.json is valid
composer validate --no-check-publish composer.production.json   # composer.production.json is valid
```

> `composer.json` 若在 12.2 刚改过 `name`、还没 `composer update --lock`，这一步会先报 lock 过期（坑 #30）——先刷 lock 再 validate。

双 composer 勾选：

```
□ composer.production.json 的 name 已改
□ 生产私包源（repositories.url + require pin）指向自己的配置
□ composer validate 两份都过
```

## 12.5 外部接线

**① moo-monitor 云端 token 置空/占位**：骨架的 `.env` 里带着**接入作者云端**的真实 token，
新项目**必须清空或换占位**，否则你的 runtime/慢 SQL 会推到别人的云端项目：

```dotenv
MOO_MONITOR_CLOUD_ENABLED=false      # 没接自己的云端前先关掉
MOO_MONITOR_CLOUD_TOKEN=             # 置空；接入自己的 moo-scaffold-cloud 项目后再填
```

> `.env.example` 里本就没有真实 token（只有 SQL 慢查询开关），所以**从 `.env.example` 拷出来的 `.env` 是干净的**；
> 但你若直接 clone 了带 `.env` 的目录（`.env` 通常被 gitignore、clone 不会带下来，但本地拷贝可能带），务必核对这两行。

**② gitee 远端**：换成你自己的仓库地址（下一步 git init 后加）。

**③ git init fresh**（12.1 已 `rm -rf .git`，这里重开一条干净历史）：

```bash
cd ..                                  # 回到 demo-shop 根
git init
git add -A
git commit -m "chore: 从 moo-engine-skeleton 起手 demo-shop"
git remote add origin git@gitee.com:<你>/demo-shop.git
```

外部接线勾选：

```
□ MOO_MONITOR_CLOUD_TOKEN 置空/占位、CLOUD_ENABLED=false
□ git init + 首个 commit（丢掉骨架历史）
□ git remote 指向你自己的仓库
```

## 12.6 装依赖与建库

**装依赖**：新项目正常走 `composer install`（会按 `composer.lock` 装齐 vendor）。

> 起手演练时若手上已有一份骨架的 `vendor/`，可以直接拷过来省一次全量下载
> （本章的 `demo-shop` 演练就是**复用骨架 `vendor/` 拷贝**加速的，如实说明）；
> 拷贝 vendor 后跑一次 `composer install`（或 `composer dump-autoload -o`）让 autoload/包发现对齐即可。正式项目建议照常 `composer install`。

**建库 + seed**：起手演练用 **sqlite** 最省事（不依赖本机 MySQL），`.env` 改两处、建个空库文件即可：

```dotenv
DB_CONNECTION=sqlite
# DB_HOST / DB_PORT / DB_DATABASE / DB_USERNAME / DB_PASSWORD 全部注释掉
```

```bash
touch database/database.sqlite
php artisan migrate --seed --force
```

> 正式项目当然用 MySQL/MariaDB（骨架 `.env.example` 默认就是 mysql + `moo_skeleton` 库），把上面五个 DB_* 填成你的真实连接即可。
> seed 会建好自建用户、角色/部门树/岗位、后台管理员（默认账号见目录页 README「默认账号」表）。

装依赖建库勾选：

```
□ composer install（或复用 vendor 后 dump-autoload）
□ .env 数据库连接（sqlite 演练 / MySQL 正式）
□ migrate --seed 成功
```

## 12.7 首次冒烟五命令

起手完成后跑这五条，全绿即「clone→改名→配 env→跑通」闭环成立：

```bash
# ① moo-system 集成自检：期望 All 5 required checks passed
php artisan moo-system check

# ② 测试全绿：骨架自带 57 passed（sqlite :memory:，不碰你的业务库）
php artisan test

# ③ 迁移状态：所有 migration 都是 Ran，没有 Pending
php artisan migrate:status

# ④ 路由清单：确认 /api/admin/authenticate、/app 等路由都在
php artisan route:list

# ⑤ 起服务，登录接口拿到 200 + token
PHP_CLI_SERVER_WORKERS=4 php artisan serve --host=127.0.0.1 --port=8088 --no-reload &
curl -s -w '\nHTTP %{http_code}\n' -X POST http://127.0.0.1:8088/api/admin/authenticate \
  -H 'Content-Type: application/json' -H 'Accept: application/json' \
  -d '{"account":"13800000000","password":"admin888"}'
```

冒烟实录（`demo-shop`）：

```
① 🎉  All 5 required checks passed. moo-system 配置健康。
② Tests:    57 passed (164 assertions)
⑤ {"data":{"user":{"id":"72146523823149056","real_name":"管理员",...},"token":"eyJ0eXAi…","expires_in":172800}}
  HTTP 200
```

> ⚠️ 登录字段是 **`account`**，不是 `username`——传 `username` 会先被 FormRequest 挡成 `422 The account field is required`（不是密码错）。
> 后台管理员默认账号 `13800000000 / admin888`（PersonnelSeeder 建，见 README 默认账号表）。`PHP_CLI_SERVER_WORKERS=4` 的原因见根 README「为什么必须多 worker」。

> `moo-system check` 五项：Auth provider FQN / admin 中间件组含 `jwt.auth.refresh` / classmap 无已删 `App\Models\System\*` /
> host 五个必需契约齐全 / `config:cache` 与 source 一致。任一 FAIL 都对应目录页坑表里的具体条目。

冒烟五命令勾选：

```
□ moo-system check → 5/5
□ php artisan test → 57 passed
□ migrate:status → 全 Ran
□ route:list → 关键路由在册
□ 登录接口 → HTTP 200 + token（账号字段是 account）
```

## 12.8 教学样例 Food 怎么处置（二选一）

骨架里的 **Food**（食品）是一整套教学样例：贯穿第 2 章（生成）、第 5 章（ACL）、第 6 章（移动端只读）、第 9 章（增量），
是「可运行的参考实现」。起手时二选一：

### 路径 (a)：保留作参考（推荐初期）

**什么都不用做。** 把 Food 当活文档留着——照着它的 Controller/Request/Resource/Model/测试的写法生成你自己的第一个模块，
写熟了再按路径 (b) 删掉。留着不影响任何东西（57 passed 里就包含它的测试）。

### 路径 (b)：删除清单（想要干净起点时）

Food **不是完全孤立**的——除了它自己的一堆文件，还有**共享测试借它的接口做通用回归守护**。
下面这份清单在 `demo-shop` 副本上**真机删过、删完 `43 passed` / `check 5/5` / 登录仍 200**，逐项照做即可。

**1. 源码目录（整目录删）**

```bash
cd engine
rm -rf app/Models/Food \
       app/Admin/Controllers/Food app/Admin/Requests/Food app/Admin/Resources/Food \
       app/Api/Controllers/Food   app/Api/Requests/Food
```

**2. 迁移（两个）**

```bash
rm -f database/migrations/*_create_foods_table.php \
      database/migrations/*_update_foods_table.php
```

**3. 专属测试（三个文件，共 11 例）**

```bash
rm -f tests/Feature/ApiFoodTest.php \        # 5 例（移动端只读 + 守卫隔离）
      tests/Feature/FoodAclTest.php \        # 5 例（动作级授权 401→403→200）
      tests/Feature/FoodIncrementalTest.php  # 1 例
```

**4. ⚠️ 共享测试 `RegressionTest.php`（坑 #31 —— 关键，别漏）**

`RegressionTest` 里有 **3 个方法借 `api/admin/food` 端点**做**通用**回归守护（幻影路由不注册 / `page_limit` 上限 / 筛选字段对齐），
它们不是「Food 专属测试」，但删了 Food 就会红。处理：删掉这 3 个方法——

```
- test_phantom_destroy_route_is_not_registered   （借 food 验「无 destroy 方法则不注册 DELETE」）
- test_food_price_filter_is_alive                （借 food 验「IndexRequest 未放行键到不了 filter」）
- test_page_limit_is_capped                      （借 food 验 page_limit 上限 422）
```

保留同文件另外 5 个与 Food 无关的方法（logout 幂等 ×2、过期 token 跨守卫续签 ×2、登录限流 ×1）。

> 这 3 条守护的**通用契约**（幻影路由、page_limit 上限）本身是好东西——等你生成了自己的第一个模块，
> 建议把它们**改指向你的模块端点**重新立起来，别让契约随 Food 一起消失。

**5. 路由（删引用与注册行，保留插入标记）**

- `routes/admin.php`：删 `use App\Admin\Controllers\Food\FoodController;`、`Route::iResource('food', …)`、`Route::post('food/{id}/toggle-status', …)`；
- `routes/api.php`：删 `use App\Api\Controllers\Food\FoodController;`、`Route::iResource('food', …)`。
- **保留** 两个文件里的 `// :insert_code_here:do_not_delete` 标记（scaffold 生成器靠它插新路由）。

**6. scaffold 样例源 + 缓存**

```bash
rm -f scaffold/database/Food.yaml scaffold/database/.snapshots/Food.yaml storage/scaffold/foods.php
rm -rf scaffold/api/admin/Food scaffold/api/api/Food
```

**7. 多语言 key（可选，cosmetic）**

- `lang/{en,zh-CN}/model.php`：Food 的枚举标签（整文件都是 Food）→ 清成 `return [];`；
- `lang/{en,zh-CN}/db.php` 与 `lang/{en,zh-CN}/validation.php` 的 `attributes`：删 `food_name/food_category/price/calories/food_status/description/stock`，保留 `id/deleted_at/created_at/updated_at` 这些通用列。

**8. ACL / 生成物（可选，理解成本高，建议先不动）**

`config/actions.php` 的 admin 段里，Food 的 ACL 动作是**不透明哈希键**（grep `food` 找不到），
`lang/*/actions.php`、`scaffold/acl/*.yaml`、`_menus_transform.yaml`、`_fields.yaml` 等也是 **moo:auth/moo:fresh 的再生成物**。
两个选择：
- **留着**（推荐）：这些是「不存在路由的孤儿条目」，运行时永远命中不到，**不影响 test / check / 运行**（本节实测就是留着它们、仍全绿）；
- **regenerate**：跑 `moo:auth` 从 Food-free 路由重建——但注意它会整文件重写 `config/actions.php`，
  **把手动合并进 whitelist 的 8 个个人中心 key 一起冲掉（坑 #25）**，重跑后必须再合并回去。不熟悉前者更省心。

**删完验证**（`demo-shop` 实录）：

```bash
composer dump-autoload -o
php artisan config:clear
php artisan migrate:fresh --seed --force
php artisan moo-system check     # 🎉 All 5 required checks passed
php artisan test                 # Tests: 43 passed （57 − 11 专属 − 3 RegressionTest 借用）
```
