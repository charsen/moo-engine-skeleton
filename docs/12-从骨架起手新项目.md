# 第 12 章　从骨架起手新项目

前面十一章解释骨架是怎样搭起来的。本章只解决一个问题：怎样把 `moo-engine-skeleton` 可靠地变成一个新的业务后端。

推荐入口是仓库根目录的 `./init-project`。它可以在没有 `.env`、数据库和 `vendor/` 的干净 clone 中运行，把原本容易遗漏的改名、密钥、ACL、静态资源和验证步骤收成一次可重复执行的初始化流程。

## 12.1 最短起手路径

```bash
git clone git@gitee.com:charsen/moo-engine-skeleton.git orders
cd orders

./init-project \
  --name=acme/orders \
  --app-name="Orders" \
  --scaffold-user=developer \
  --fresh-git
```

命令完成后会输出本次 scaffold 账号及密码。密码只显示在终端，不写入 Git；请立即存进团队的密码管理器。

查看完整参数：

```bash
./init-project --help
```

常用选项：

| 参数 | 作用 |
| --- | --- |
| `--keep-demo` | 保留 Food 教学模块；默认删除，输出干净业务项目 |
| `--keep-tutorial` | 保留教程、plans、交接资料；默认只留下项目自己的文档 |
| `--with-frontend` | 额外执行 `npm install && npm run build` |
| `--fresh-git` | 把骨架 `.git` 可恢复地归档到项目同级目录，再初始化项目自己的 Git 历史 |
| `--no-commit` | 与 `--fresh-git` 配合，只初始化 Git，不创建首个 commit |
| `--force` | 明确允许重新初始化；会清空 SQLite 数据和本地 scaffold 账号，使用前先确认都可重建 |

## 12.2 初始化器实际完成什么

初始化器按顺序完成以下工作，任一步失败都会返回非零状态并停在明确的错误位置：

1. 校验 PHP 8.2+、Composer 与项目名。
2. 同步修改 `composer.json`、`composer.production.json`、`.env.example` 和 `.env` 的项目身份。
3. 创建 SQLite 数据库，使首次 Composer 脚本在数据库尚未部署时也能启动 Artisan。
4. 删除 scaffold 运行历史、监控运行时；默认删除 Food 源码、迁移、路由、测试和语言残留。
5. 分别刷新开发与生产 lock 的 content hash，再按开发 lock 安装依赖。
6. 生成全新的 `APP_KEY`、`JWT_SECRET`；移除 demo 后先清理可能残留的 config/route cache，再按当前路由重建 admin/api ACL，并恢复个人中心白名单。
7. 执行 migrate/seed，发布 scaffold 静态资源并创建本地 scaffold 账号。
8. 默认移除教程与内部计划，写入项目自己的 README、协作说明、notes 和 `.moo-project.json`。
9. 运行 `moo-system check`、测试、迁移状态、路由列表及两套 Composer 校验。
10. 按需归档骨架 Git 历史并建立项目自己的首个提交。

初始化器不会替你决定生产数据库、Redis、域名、私包授权和 Snowflake 多机编号。这些属于部署环境，完成初始化后仍须按 `DEPLOY-CHECKLIST.md` 逐项配置。

## 12.3 为什么先用 SQLite

`.env.example` 默认使用 SQLite，是为了让新 clone 不依赖本机 MySQL/MariaDB 就能完成安装和自检。正式部署前再把 `.env` 切到真实数据库：

```dotenv
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=orders
DB_USERNAME=orders
DB_PASSWORD=<secret>
```

Snowflake 参数也要在生产环境明确配置：

```dotenv
SNOW_FLAKE_DATA_CENTER_ID=1
SNOW_FLAKE_WORKER_ID=1
SNOW_FLAKE_START_TIME=2026-01-01
```

- `start_time` 一旦在线上生成过 ID 就不要再改。
- 多实例部署时，每台机器的 `(DATA_CENTER_ID, WORKER_ID)` 组合必须唯一。

## 12.4 开发与生产 Composer 完全分离

骨架维护两套配置和两把 lock：

| 环境 | 配置 | lock |
| --- | --- | --- |
| 本地开发 | `engine/composer.json` | `engine/composer.lock` |
| 生产部署 | `engine/composer.production.json` | `engine/composer.production.lock` |

生产部署使用 Composer 原生的 `COMPOSER` 环境变量，不再复制或覆盖 `composer.json`：

```bash
cd engine
COMPOSER=composer.production.json composer install --no-dev --optimize-autoloader
```

`pull.sh --production` 已内置同一机制。这样生产工作树不会长期出现 `M engine/composer.json`，开发 lock 也不会被误拿来校验生产约束。

骨架当前生产约束为：

- `charsen/moo-scaffold ^2.1.3`
- `charsen/moo-monitor-laravel ^0.1`
- `charsen/moo-system ^1.6.17`

新项目首次上线前应确认这些版本和仓库来源仍符合自己的授权与发布策略。

两套配置都应通过校验和审计：

```bash
cd engine
composer validate --no-check-publish composer.json
composer audit --locked
COMPOSER=composer.production.json composer validate --no-check-publish
COMPOSER=composer.production.json composer audit --locked
```

## 12.5 Food 教学模块怎么选

默认不加 `--keep-demo`，初始化器会删除 Food 教学模块，并按剩余路由重建 ACL。这是新业务项目的推荐状态。

第一次接触这套骨架、希望对照完整 CRUD 写法时，可以保留样例：

```bash
./init-project --name=acme/orders --keep-demo
```

Food 贯穿生成、ACL、双守卫和增量开发章节，适合作为活文档；但不要把它误当成项目业务域。熟悉后可以重新从骨架初始化一个不带 demo 的正式项目。

## 12.6 初始化后的验收

初始化器已经自动执行以下命令，日后升级依赖或调整骨架时也应把它们作为质量门禁：

```bash
cd engine
php artisan moo-system check
php artisan test
php artisan migrate:status
php artisan route:list --except-vendor
composer audit --locked
./vendor/bin/pint --test
```

原始骨架当前基线为 `64 passed (230 assertions)`；默认移除 Food 后本轮实测为 `49 passed (161 assertions)`，保留 Food 教学模块则为 `64 passed (230 assertions)`，以初始化器最终全绿为准。`moo-system check` 应为 5/5。

再做一次真实 HTTP 登录冒烟：

```bash
PHP_CLI_SERVER_WORKERS=4 php artisan serve --host=127.0.0.1 --port=8088 --no-reload
```

另开终端：

```bash
curl -s -w '\nHTTP %{http_code}\n' \
  -X POST http://127.0.0.1:8088/api/admin/authenticate \
  -H 'Content-Type: application/json' \
  -H 'Accept: application/json' \
  -d '{"account":"13800000000","password":"admin888"}'
```

期望返回 token 与 `HTTP 200`。登录字段是 `account`，不是 `username`。

scaffold 控制台位于 `/scaffold`。其账号由初始化器创建，保存在本地忽略文件中；换机或重新 clone 后需要重新创建。

## 12.7 Git 与资料边界

使用 `--fresh-git` 时，初始化器不会直接删除骨架历史，而是把 `.git` 移到项目同级的隐藏备份目录，再创建新仓库。确认新项目稳定并已推送后，你可以自行决定是否保留该备份。

默认生成的业务项目不会带走：

- 教程 `docs/`
- 内部计划 `plans/`
- 历史交接 `HANDOFF.md`
- scaffold API 发布历史和运行时转储
- Food 教学业务

需要把新发现反哺到骨架时，在骨架仓库修复并补测试，不要把某个业务项目的客户信息、凭据、域名或运行时数据复制回来。

至此，新项目已经拥有独立身份、密钥、数据库、ACL、依赖锁、scaffold 账号和 Git 历史。后续加表、加接口、同步 ACL 与测试，继续参考[第 9 章 日常增量开发](./09-增量开发工作流.md)。
