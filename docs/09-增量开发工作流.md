# 第 9 章　日常增量开发：改表与加接口

目标：在已上线的 `foods` 表上走一遍**真实的日常改动**——加一个 `stock`（库存）字段、
加一个 `toggleStatus`（上架/下架切换）自定义接口，再给空架子的移动端 `Api/` 分片
生成第一个只读业务接口（9.8），全程用增量命令完成并真机验证。

---

## 9.1 为什么增量工作流值得单独一章

前面八章是「绿地」叙事：`moo:free` 一条命令把一个实体的约 15 个文件全部生成出来。
但 `moo:free` 在每个实体的生命周期里**只用一次**——真实开发 90% 的时间是在已有的表上
改字段、加接口。这时候的命令组合完全不同（`moo:migration` / `moo:adder` / `moo:i18n` /
`moo:auth` / `moo:api`），而且会撞上绿地阶段感受不到的核心问题：

> **哪些文件生成器会自动覆盖、哪些必须自己手动补？**

搞不清这个边界，要么手改了「再生成区」被下次生成无声抹掉，要么漏补了「生成一次」
的文件让新字段静默失效。本章就用一次完整实操把这条边界画清楚。

**前提**：本章在仓库**最终态**上演练，需要第 7 章完成后的状态（moo-system 已接入、
ACL 已开、`13800000000` / `admin888` 可登录），开发服务器按惯例跑在 8088：

```bash
PHP_CLI_SERVER_WORKERS=4 php artisan serve --host=127.0.0.1 --port=8088 --no-reload
```

## 9.2 加字段：yaml → moo:fresh → moo:migration

### ① 真相源先行：编辑 yaml

一切结构改动从 `scaffold/database/Food.yaml` 开始（**不要**直接写迁移）。
在 `fields:` 里 `price` 之后加一行，沿用行内 map 风格：

```yaml
price: { name: 价格, type: int, default: 0, desc: '单位：分' }
stock: { name: 库存, type: int, default: 0 }
```

### ② moo:fresh 重建缓存

改完任意 schema yaml 后必须跑（第 2 章讲过，所有生成器都读这份缓存）：

```bash
php artisan moo:fresh
```

```
🔍 ./scaffold/database/Food.yaml .... Parsed
🔄 ./scaffold/models.php / model_ids.php / controllers.php / tables.php
   / enums.php / foods.php / database/_fields.yaml / fields.php .... Updated
SUCCESS ✅ Completed.
```

注意：`moo:fresh` 只重建 `storage/scaffold` 缓存与 `scaffold/` 下的派生文件，
**不碰**任何 `app/` 下的代码（实测前后 `app/Models/Food/Traits/FoodTrait.php`、
`Enums/*.php` 的 md5 完全一致）。代码层面的覆盖发生在后面的 `moo:model`。

### ③ moo:migration 生成增量迁移

```bash
php artisan moo:migration --help
# Usage: moo:migration [<schema_name>]   （Ex: System）

php artisan moo:migration Food
```

```
ℹ️  1 个 migration 文件已生成
  + engine/database/migrations/2026_06_11_143750_update_foods_table.php

提示: scaffold 不再自动 git commit。请手动:
  git add scaffold/database/Food.yaml scaffold/database/.snapshots/Food.yaml database/migrations/...

 💬 Run 'artisan migrate' now (yes/no) [yes]:
 > 🎯 Running migrate
  2026_06_11_143750_update_foods_table .... 38.04ms DONE
```

三个要点：

1. **生成的是增量 `Schema::table`，不是重建表**——核心内容：

   ```php
   public function up(): void
   {
       Schema::table('foods', static function (Blueprint $table) {
           $table->integer('stock')->unsigned()->default(0)->comment('库存')->after('price');
       });
   }

   public function down(): void
   {
       Schema::table('foods', static function (Blueprint $table) {
           $table->dropColumn('stock');
       });
   }
   ```

   连 `->after('price')` 都按 yaml 里的字段顺序排好了；`int` + `default: 0` 推导成
   `unsigned()->default(0)`，注释取 yaml 的 `name`。它怎么知道「只加了 stock」？
   diff 的基准是 `scaffold/database/.snapshots/Food.yaml` 快照（数据库设计器用的同一套
   diff + writer）。**快照 + yaml + 迁移要一起提交**，否则下次 diff 基准错乱。

2. **末尾追问 `Run 'artisan migrate' now (yes/no) [yes]`，默认 yes**——非交互管道下
   会直接取默认值把迁移跑了，脚本里调用时要心里有数。跑完用
   `SHOW COLUMNS FROM foods` 确认：`stock int(10) unsigned NOT NULL DEFAULT 0`，
   位置正好在 price 之后。

3. **可以放心重复执行**：纯枚举标签改动（见 9.3 的 i18n 一节）后再跑一次
   `moo:migration Food`，输出 `ℹ️ 无变更,跳过生成 migration。`——它只对结构性变更
   产出迁移。

## 9.3 手补「生成一次」的文件（本章核心）

数据库有 stock 了，但代码层还不认识它。第 2 章提过 scaffold 的两类文件——
「再生成区」（每次生成都覆盖）和「只生成一次」（已存在就跳过）——增量开发时
这条边界决定了你的手工清单。

### 9.3.1 先实测：再生成区会不会自动跟上？

```bash
php artisan moo:model Food
```

```
💤 ./app/Models/Food/Food.php ............... Model already exists   ← 生成一次，跳过
🔄 ./app/Models/Food/Traits/FoodTrait.php ............... Updated   ← 再生成区，覆盖
🔄 ./app/Models/Food/Enums/FoodCategory.php ............. Updated
🔄 ./app/Models/Food/Enums/FoodStatus.php ............... Updated
```

实测两点：

- `Food.php`（Model 本体）**只生成一次**，已存在就 💤 跳过——`$fillable` / `$attributes`
  这些必须自己补（下一小节）。
- `Traits/FoodTrait.php`、`Enums/*.php` 是**再生成区，被无条件覆盖**。本次覆盖后与
  已提交版本的 diff 只剩代码风格差异（生成器输出 `<?php declare(strict_types = 1);`
  单行风格、`(int)$x` 不带空格；仓库里的版本被 pint 格式化过）。9.6 跑完 pint 后这些
  文件与提交版本**字节级一致**——即「再生成区被覆盖 + pint 收尾 = 无净变更」
  （stock 不是枚举字段，不会新增 `getStockTxtAttribute` 之类访问器）。
  **结论：重生成再生成区永远是安全的，后面跟一次 pint 即可。**

### 9.3.2 手补 Model（`app/Models/Food/Food.php`，3 处）

```php
 * @property int $stock 库存            ← docblock，放 price 之后
'stock' => 0,                           ← $attributes 默认值
protected $fillable = [..., 'price', 'stock', 'calories', ...];   ← 批量赋值白名单
```

漏改 `$fillable` 的症状最阴险：create/update 接口照样返回 201/200，但 stock 被
批量赋值**静默丢弃**，库里永远是默认值。

### 9.3.3 手补 Request（`app/Admin/Requests/Food/Food/{Store,Update}Request.php`，各 2 处）

两个文件改法相同：

```php
// rules() 里 price 之后：
'stock' => ['nullable', 'integer', 'min:0'],

// formLayout() 里 price 之后：
['stock'],
```

`formLayout()` 不补的话，`/scaffold` 调试器和前端的 create/edit 表单不会出现该控件。
IndexRequest 的筛选支持本次未加（可选项；要加的话还得同步 ModelFilter）。

### 9.3.4 手补 Controller 侧硬编码字段清单（最容易漏的两处）

- `app/Admin/Controllers/Food/Traits/FoodTrait.php`——注意 **controller 的 trait 是
  「生成一次」的，和 Model 的同名 trait（再生成区）不是一类**！`getListFields()` 的
  `$fields` 数组、`getListColumns()` 的 `$columns` 数组各加 `'stock'`。
  漏掉的症状：列表接口查不出 / 表头不显示新字段。
- `app/Admin/Controllers/Food/FoodController.php` 的 `show()`：`$fields` 和 `$columns`
  两个硬编码数组各加 `'stock'`。

### 9.3.5 i18n 同步：`moo:i18n`（注意：不接参数）

```bash
php artisan moo:i18n Food
# 报错：No arguments expected for "moo:i18n" command, got "Food".
php artisan moo:i18n          # ← 正确用法，全量同步
```

```
🔄 ./lang/{en,zh-CN}/model.php / validation.php / db.php .... Updated
```

stock 的标签进了 `db.php`（`'stock' => '库存'/'Stock'`）和 `validation.php` 的
attributes。

> ⚠️ **坑 #23**：`lang/*/model.php` 也是**再生成区**。本仓库的 `lang/en/model.php`
> 枚举标签曾被手工首字母大写过（'Fruit'、'On Shelf'），而 yaml 里写的是小写
> （`fruit: [1, fruit, 水果]`）——`moo:i18n` 一跑直接回退成小写。**正确修法是改真相源**：
> 把 yaml 枚举第二个元素改成 `[1, Fruit, 水果]` 等，再 `moo:fresh` + `moo:i18n`，
> diff 即恢复干净（lang 目录最终只剩 4 个文件各 +1 行 stock）。
> 教训：**英文标签想要什么样，写进 yaml，别改 lang 文件。**

## 9.4 `moo:adder` 加自定义 action：toggleStatus

CRUD 之外的接口（如「上架/下架切换」）用 `moo:adder` 增量追加——**别重跑 `moo:free`**。

### 9.4.1 它是交互式命令，stdin 可以管道喂

`moo:adder --help` 只有两个可选参数 `app`、`folder`，其余全靠交互 prompt，
实测链路（answers 从 stdin 喂，每行一个回答）：

1. `Choose a controller:`（列出 `<NEW_ONE>` 和现有控制器，输编号）；
2. `Enter: {action} {request} {resource}:`（空格分隔；request / resource 填 `null`
   表示不要——不要 request 就不生成 FormRequest 形参，resource 为空默认 `BaseResource`）；
3. `Enter: {method} {url}:`（如 `post food/{id}/toggle-status`；留空则不加路由）。

> ⚠️ **坑 #24**：folder 参数**必须带尾部斜杠**。写 `Food` 会把 folder 与控制器文件名
> 直接拼接（缺 `/`），候选显示成 `FoodFoodController`，随后直接崩：
>
> ```bash
> printf '1\ntoggleStatus null null\npost food/{id}/toggle-status\n' | php artisan moo:adder admin Food
> # file(.../app/Admin/Controllers/FoodFoodController.php): Failed to open stream
> ```

正确写法（`Food/` 带斜杠）：

```bash
printf '1\ntoggleStatus null null\npost food/{id}/toggle-status\n' | php artisan moo:adder admin Food/
```

```
💬 Choose a controller:  [0] <NEW_ONE>  [1] Food/FoodController
🎯 Running Controller Adder
  ➕ ./app/Admin/Controllers/Food/FoodController.php ... @toggleStatus():
  ➕ ./routes/admin.php ............ Route post food/{id}/toggle-status
```

### 9.4.2 adder 只产空壳 + 路由，业务逻辑自己填

它在控制器**末尾**追加了一个空壳：

```php
/**
 * toggleStatus
 */
public function toggleStatus(): BaseResource
{
    $result = [];
    return BaseResource::make($result);
}
```

路由插进了 `routes/admin.php` 的 JWT 登录 group 内、`// :insert_code_here:do_not_delete`
标记之前（位置正确），但用的是 FQCN：

```php
Route::post('food/{id}/toggle-status', [App\Admin\Controllers\Food\FoodController::class, 'toggleStatus']);
```

手工收尾三件事：

**① 路由美化**：改用文件头已 `use` 的短类名 `[FoodController::class, 'toggleStatus']`，
去掉多余空行。

**② 空壳改成真实现**（注意 stub 不带 `$id` 形参，要自己加）：

```php
/**
 * 上架/下架切换（权限复用 update，见 $transform_methods）
 */
public function toggleStatus(int|string $id): BaseResource
{
    $result = $this->model->findOrFail($id);
    $result->food_status = $result->food_status === FoodStatus::ON_SHELF->value
        ? FoodStatus::OFF_SHELF->value
        : FoodStatus::ON_SHELF->value;
    $result->save();

    return BaseResource::make($result);
}
```

头部加 `use App\Models\Food\Enums\FoodStatus;`。按本生态约定枚举**不进 `$casts`**
（坑 #19 的教训），字段是裸 int，比较时用 `->value`。

**③ 权限接线**：类属性里加

```php
protected array $transform_methods = [
    'toggleStatus' => 'update',
];
```

**为什么用 transform 复用 update 权限，而不是 `@acl` 注册新权限？**
「切换上下架」本质上就是一种更新——能改 food 的人就该能切它的状态，
没必要在授权页多出一个独立勾选项。这正是基类默认映射 `restore => trashed` 的同款纹理，
而那些方法的 docblock 也**不带 `@acl`**：二者并存不自洽——挂了 transform 的 action，
`checkAuthorization()` 永远按映射目标（update）鉴权，这时再写 `@acl` 只会让授权页
多出一个**勾了也不生效的死条目**。所以：transform 复用权限的 action 不写 `@acl`，
复用关系写在 docblock 注释里即可。

## 9.5 ACL 与接口文档同步

路由和权限映射变了，把两份派生物刷新（都是再生成区，放心重跑）：

```bash

```

```

```

验证 `scaffold/acl/admin.yaml`：toggleStatus 被正确识别为权限复用——


> ⚠️ **坑 #25**：被重写的不只是注释——**第 7 章手动加进 `config/actions.php` 白名单的
> 个人中心 8 个 key（坑 #20）也会被冲掉**。moo:auth 只把「docblock 不带 `@acl` 的

> （属可授权动作），不在自动产出里。每次重跑 `moo:auth` 后都要把这 8 个 key
> **合并回 whitelist**，否则零授权角色又被锁死在门外。本仓库在 `FoodAclTest` 的
> 个人中心用例里加了守护断言：白名单丢 key，测试直接红。

## 9.8 移动端分片的第一个业务接口（只读 Food）

第 6 章搭好了 `Api/` 分片的骨架（AuthController + user 守卫 + 双向隔离），但它至今
是个空架子——一个业务接口都没有。本节给它加第一个：**Food 只读（index + show）**，
顺带演练生成器在 api 分片上的行为（和 admin 分片**不完全一样**）。

### 9.8.1 yaml 的 `app` 数组就是分片开关

一张表要在哪些分片出接口，开关在 `scaffold/database/Food.yaml` 的 `controller.app`：

```yaml
controller:
    app: ['admin', 'api']     # ← 原来只有 'admin'，加上 'api'
    class: FoodController
```

老规矩 `moo:fresh` 重建缓存，然后跑 api 分片的生成（`moo:free` 可以按分片重复跑，
admin 分片已存在的文件全部 💤 跳过）：

### 9.8.2 裁剪成移动端只读

stub 虽然只给了 index/show，但内容还是后台味的：列表带 `columns` / `form_widgets`
（前端后台表格/表单的元数据）、`->append(['options'])`（每行的回收站动作列表）、
show 的字段白名单里有 `deleted_at`。移动端统统不需要——它只消费数据：

- **index**：保留 `filter + paginate`（`IndexRequest` 的 `food_name` / `food_category` /
  `food_status` 筛选白名单照用），返回裸的 `BaseResource::collection($result)`——
  分页 meta `{page, per_page, total, total_page}` 是 `BaseResourceCollection` 自带的；
- **show**：字段白名单去掉 `deleted_at`（移动端没有回收站概念），也不 `withTrashed()`
  ——软删的食品对移动端就是 404；形参对齐生态约定用 `int|string $id`（雪花字符串主键）；
- `Controllers/Food/Traits/FoodTrait.php`（「生成一次」区，是你的）裁剪到只剩
  `getListFields()`；`getListColumns` / `getListFormWidgets` 连同 import 一起删。

**为什么不用删路由？** 第 3 章注册的 `iResource` 宏是**按控制器方法反射注册**的——
方法不存在路由就不存在。裁完 `route:list` 验证：

```bash
#  GET|HEAD  app/food/{id}     app.food.show
```

## 9.9 给实体配专属 Resource：链式字段控制

到目前为止 Food 全程裸用 `BaseResource`——它的 `toArray()` 把模型属性原样倒出来。
而生态宣传的 `->show()/->hide()/whenDate()/whenTrashed()` 链式字段控制，要发挥威力
得有一个**每实体的专属 Resource** 当展示面。本节给 admin 分片配一个 `FoodResource`，
只动 show 与 trashed 两个 action（index 保持裸 `BaseResource` 不动，控制示例面）。

### 9.9.1 生成器实录：`moo:resource` 的「静默成功」（坑 #27）

scaffold 自带 Resource 生成器，先试它：

```bash
php artisan moo:resource Food
#    SUCCESS  ✅ Completed.        ← 但 app/Admin/Resources/ 根本没出现，一个文件都没生成！
```

原因（读 `moo-scaffold/src/Generator/CreateResourceGenerator.php` 的
`getResourceTargets()`）：生成器**只为 yaml 里 `controller.resource` 声明过的分片**
产文件。Food.yaml 一直只写了 `controller.app: ['admin','api']`，编译缓存里的
`resource` 数组是空的 → 0 个输出目标，循环空转，命令照样报 SUCCESS。
这正是坑 #26 的另一面：api stub 引用 `FoodResource`，可默认根本没人生成它——开关在 yaml：

```yaml
        controller:
            app: ['admin', 'api']
            resource: ['admin']      # ← moo:resource 的输出开关，按分片声明（作者生产项目同款写法）
            class: FoodController
```

本节只开 `admin`（api 分片在 9.8.2 已定为裸 `BaseResource`，不回头扩面）。
yaml 是真相源，改完照例 `moo:fresh`，再生成就有产物了：

```bash
php artisan moo:fresh
php artisan moo:resource Food
#  ✨ ./app/Admin/Resources/Food/FoodResource.php ..................... Created
```

输出目录来自 `config/scaffold.php` 的 `controller.admin.resource_path`
（`app/Admin/Resources/`），HTTP 分片自此「四件套」补齐成 Controllers / Requests /
Resources 三兄弟目录。Resource 和 Model/Controller 一样是「生成一次」区
（已存在则跳过，`-f` 才覆盖）——业务字段映射写在这里是安全的。

### 9.9.2 生成的 FoodResource 与四个条件方法

生成的类直接可用，骨架只做了两处教学微调（`whenDate` 写明格式、`whenTrashed`
里把 Carbon 格式化），完整版在
`engine/app/Admin/Resources/Food/FoodResource.php`：

```php
class FoodResource extends BaseResource
{
    public function toArray(Request $request): array
    {
        $data = collect([
            'id' => $this->id,
            'food_name' => $this->food_name,
            'food_category' => $this->whenHas('food_category'),
            'food_category_txt' => $this->whenAppended('food_category_txt'),
            // ... price / stock / calories / food_status(_txt) / description 同理
            'deleted_at' => $this->whenTrashed($this->deleted_at?->format('Y-m-d H:i')),
            'created_at' => $this->whenDate('created_at', 'Y-m-d H:i'),
            'updated_at' => $this->whenHas('updated_at'),
            'options' => $this->whenAppended('options'),
        ]);

        // 最后过 filterFields()，调用方仍可链 ->show('id,food_name') / ->hide('price') 二次裁剪
        return $this->filterFields($data);
    }
}
```

四个条件方法（缺席的键不会出现在 JSON 里，靠 `MissingValue` 实现）：

| 方法 | 来源 | 触发条件 | 生成器的默认选型 |
|---|---|---|---|
| `whenHas('x')` | Laravel JsonResource | 查询 select 了该列 | 非索引字段全用它（窄 select 列表不出 null 噪音） |
| `whenAppended('x')` | Laravel JsonResource | `->append(['x'])` 过 | 枚举 `_txt` 文案、`options` 动作列表 |
| `whenDate('x', fmt)` | BaseResource | 列存在 | 日期列统一格式化，默认 `'Y-m-d H:i'` |
| `whenTrashed($v)` | BaseResource | 链过 `->trashed()` | `deleted_at` 只在回收站语境出现 |

收尾的 `filterFields()` 别删——`->show()/->hide()` 这对链式裁剪就靠它落地。
适用场景一句话：**`show()` 给轻量端点白名单瘦身，`hide()` 临时抹掉个别敏感列，
都不必为此写第二个 Resource 类。**

### 9.9.3 接入 Controller：show 与 trashed

`FoodController` 两处替换（import 补 `use App\Admin\Resources\Food\FoodResource;`）：

```php
// trashed()：集合上链 ->trashed()，传播给每个 item 的 whenTrashed
return FoodResource::collection($result)
    ->trashed()
    ->additional([...]);

// show()：详情查的是 withTrashed，已软删的记录链 ->trashed(true) 让 deleted_at 一并输出
return FoodResource::make($result)
    ->trashed($result->trashed())          // SoftDeletes 的 trashed() 返回 bool
    ->additional(['columns' => ColumnsCollection::make($columns)]);
```

集合链能成立是因为 `BaseResourceCollection` 和 `BaseResource` 共用
`BaseResourceTrait`，`processCollection()` 会把集合上的 `trashed/hide/show`
状态逐个传给底下的 Resource。

```bash

./vendor/bin/pint app/Admin/Resources app/Admin/Controllers/Food
```
