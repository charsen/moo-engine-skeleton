# 第 8 章　部署上线（可选）

目标：把骨架部署到一台 Linux 服务器（nginx + php-fpm），并避开两个只在生产才会
暴露的坑：**雪花 ID / JWT 黑名单的共享缓存**、**清缓存导致已注销 token 复活**。

> 本章是通用部署流程的"骨架特化版"——只讲本项目特有的注意点，
> nginx/MySQL 的基础安装请参考各发行版文档。

---

## 8.1 服务器前提

- PHP 8.2+（含 `pdo_mysql / mbstring / bcmath / redis` 扩展）+ Composer；
- MySQL 8 / MariaDB 10.6+，Redis；
- nginx + php-fpm；git（按 tag 拉代码）；certbot（HTTPS 证书，见 8.4）。

### 首次部署系统依赖前置（用部署脚本才需要）

仓库根的部署脚本（`pull.sh` / `cache.sh` / `backup.sh`，见 8.8）对系统命令有硬依赖，
**新机器第一步先装齐**，否则 `pull.sh` 第一屏就 `exit 1`：

```bash
# Debian / Ubuntu
sudo apt install -y git openssh-client jq util-linux    # util-linux 含 flock
# CentOS / RHEL / Alma
sudo yum install -y git openssh-clients jq util-linux
```

- `git` / `ssh` / `composer` / `jq`：`pull.sh` 启动 `require_command` 这四件，缺任一直接罢工
  （`jq` 解析私包 manifest，`ssh` 验私包拉取权限）。
- `flock`（`util-linux`）：缺了不报错、但失去部署并发保护（macOS 默认无，脚本会 WARN 跳过）——
  Linux 生产强烈建议装。
- **sudo `secure_path` 坑**：`sudo` 会把 `PATH` 重置成 `secure_path`（默认不含 `/usr/local/php/bin`
  等运维手装的 PHP 路径），导致 `composer` 走了系统老版 PHP。`tools/_common.sh` 顶部的 PATH 守卫会补
  几个常见位置兜底，但你的 PHP 若装在别处，仍要 `which php && php -v` 核对版本对不对。
- 装 `jq` 时若弹 needrestart dialog（Debian/Ubuntu），Tab → 选 `<Cancel>`，避免它顺手重启 php-fpm。

> 完整首次部署核对单见仓库根 [`DEPLOY-CHECKLIST.md`](../DEPLOY-CHECKLIST.md) 的 **O-0** 段。

## 8.2 取码与依赖（Packagist + moo-system VCS）

> `moo-scaffold`、`moo-monitor-laravel` 是开源包，目标状态下部署时直接从 Packagist 解析；
> `moo-system` 是商业包，仍通过 `git@gitee.com:charsen/moo-system.git` 的 VCS 仓库安装。
> 部署机只需要提前配置能读取 `moo-system` 仓库的 SSH key。
> 当前过渡期若 Packagist 尚未同步开源包目标版本，`composer.production.json` 只是目标样例，
> 不能不改就照抄部署；生产 composer 需要临时保留 scaffold / monitor 的 VCS 仓库配置，
> 发布同步完成后再删掉它们。

```bash
git clone <你的仓库> moo-engine-skeleton && cd moo-engine-skeleton/engine
git checkout <发布 tag>

# Packagist 目标版本可解析后，可切到生产 composer：
cp composer.production.json composer.json

# 首次部署或 composer.json 与 lock 不一致时，用 update 解析并生成 lock；
# 后续可提交生产 lock，再改用 composer install --no-dev --optimize-autoloader。
composer update --no-dev --optimize-autoloader
```

> 想要可复现构建：第一次部署成功后，把生成的 lock 另存为
> `composer.production.lock` 进仓库，以后部署改为
> `cp composer.production.json composer.json && cp composer.production.lock composer.lock && composer install --no-dev`。

> 包的获取凭据：目标状态下开源包从 Packagist 获取，不需要部署公钥；`composer.production.json`
> 只给 `moo-system` 配置 Gitee VCS 仓库，所以部署机只需按授权配置 `moo-system`
> 的只读 SSH 部署公钥。

### 目录权限（漏了这步，首个请求就 500）

php-fpm 以 `www-data` 运行，clone 出来的目录归属却是你的部署账号：

```bash
# Laravel 标配的两处可写目录
chown -R www-data:www-data storage bootstrap/cache
chmod -R ug+rwX storage bootstrap/cache

# 监控缓冲（运行时异常 / 慢 SQL）落盘在 storage/moo-monitor/，上面的 storage 权限已覆盖。
# 如果生产环境先跑命令创建了 root-owned 文件，再补一次归属：
chown -R www-data:www-data storage/moo-monitor
```

### 用部署脚本代替手动 composer（推荐）

上面的手动 `cp composer.production.json composer.json && composer update` 是**为了让你看清每一步在做什么**。
日常部署推荐用仓库根的 `pull.sh` 一键完成——它把「切 production composer + 私包权限验证 +
`composer install/update` + `vendor:publish` + 清缓存修权限」串成一条：

```bash
cd /opt/moo-engine-skeleton
sudo sh pull.sh --production      # 首次部署（.env 未建，显式声明生产）
sudo sh pull.sh                  # 后续日常（自动判 production）
```

`pull.sh` 从 `engine/composer.production.json` 的 `extra."moo-private-packages"` manifest 动态解析私包列表，
**加/减包只改 manifest、不动脚本**。它与 `cache.sh` 职责严格分离（网络层 vs 本地层），退出码 `0/1/3/4`
供 cron 监控区分「主体成功但收尾失败」。脚本清单见 [`SCRIPTS.md`](../SCRIPTS.md)，私包 deploy key 配置与
双 composer.json 机制见 [`PRIVATE-COMPOSER-PACKAGES.md`](../PRIVATE-COMPOSER-PACKAGES.md)，上线核对单见
[`DEPLOY-CHECKLIST.md`](../DEPLOY-CHECKLIST.md)。

> 生产侧 `git status` 会**长期显示 `M engine/composer.json`**（Step 4 把 production 配置 cp 覆盖了它）——
> 这是**预期状态**，`pull.sh` 已自动过滤该条不 abort；**绝不要** `git commit engine/composer.json`（会污染开发版）。

## 8.3 生产 `.env` 要点

先创建 `.env` 并把数据库准备好（没有 `.env` 文件，下面的 `key:generate` 无处可写；
没建库，`migrate` 第一步就失败）：

```bash
cp .env.example .env
# 在 MySQL 里先建库（库名与 .env 的 DB_DATABASE 一致，示例用 moo_skeleton）：
#   CREATE DATABASE moo_skeleton CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

然后改这些键：

```dotenv
APP_ENV=production        # /scaffold 开发 UI 自动禁止写操作（机制见下方说明）
APP_DEBUG=false           # 必须关——异常堆栈里有路径和源码
APP_URL=https://你的域名   # 协议要与 nginx 实际监听一致，HTTPS 见 8.4

DB_DATABASE=moo_skeleton  # 连同 DB_USERNAME / DB_PASSWORD 换成生产值
DB_USERNAME=…
DB_PASSWORD=…

CACHE_STORE=redis         # ⚠ 必须：雪花 ID 跨 worker 共享、JWT 黑名单都存在 cache 里
QUEUE_CONNECTION=redis    # 操作日志/登录记录走队列，配 worker（见 8.6）
OPERATION_LOG=true
```

> ⚠ `.env.example` 里的 `REDIS_*` 几行默认是**注释掉的**（本地教程用不到）。
> 换 redis 时记得放开并填上真实连接——否则 worker 连不上 Redis，
> 8.7 冒烟项 7 会静默失败。

> **Redis 客户端与限流（0-2）**：骨架已 `composer require predis/predis`（纯 PHP 客户端，
> 无需装 `ext-redis` 也能跑；装了 phpredis 扩展则更快，二选一即可）。另外 `bootstrap/app.php`
> 只在 `APP_ENV=production` 调 `throttleWithRedis()`——生产的 `throttle:*` 限流计数器改用 Redis 原子计数，
> 多 php-fpm / 多机 worker 下才是同一份真值（单机默认 cache store 各算各的，限流形同虚设）。
> 本地与测试不触发这条，跑测试无需 Redis。

> **`APP_ENV=production` 到底"禁"了什么？** 写保护由 `EnforceScaffoldWritable`
> 中间件实现——production（或 readonly）模式下，`/scaffold` 的 db/designer、
> accounts、config、cloud/push 等写操作会被拒。而 **`moo:*` 命令与 APP_ENV 无关**：
> 它们一律 console-only（`ScaffoldProvider` 按 `runningInConsole()` 注册，HTTP
> 上下文任何环境都不注册），生产服务器的 CLI 里**照常可用**——8.5 在生产跑
> `moo:cloud:push` 靠的正是这一点。

首次部署生成密钥：

```bash
php artisan key:generate --force
php artisan jwt:secret --force     # ⚠ 只在首次！之后再换 = 所有已发 token 失效、全员重登
php artisan migrate --force
php artisan db:seed --force        # 按需：首批账号
php artisan vendor:publish --provider="Mooeen\Scaffold\ScaffoldProvider" --tag=public --force
```

> "换 `JWT_SECRET` = 全员重登"是同一行为的两面：日常**意外**换掉是事故；
> 而 8.5 坑 #22 的安全敏感场景里**主动**换，正是拿"全员重登"当手段。两处说法不矛盾。

## 8.4 nginx 站点

```nginx
server {
    listen 80;
    server_name 你的域名;
    root /var/www/moo-engine-skeleton/engine/public;
    index index.php;

    location / { try_files $uri $uri/ /index.php?$query_string; }

    # 🔒 /storage 目录防执行（放在通用 PHP location 之前，先匹配先生效）
    location ~* ^/storage/.*\.(php[0-9]*|phtml|phar|cgi|pl|py|sh)(/.*)?$ { return 404; }

    location ~ \.php$ {
        fastcgi_pass unix:/run/php/php8.2-fpm.sock; # 按服务器实际 PHP-FPM 版本调整
        fastcgi_index index.php;
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
    }
}
```

### `/storage` 目录防执行（安全加固）

`public/storage` 是 `storage/app/public` 的软链接——**用户上传的文件都落在这里**。如果攻击者上传一个
`evil.php` 并能直接 `GET /storage/evil.php`，nginx 的通用 `location ~ \.php$` 会把它交给 php-fpm **执行**，
等于任意代码执行。上面那条 `location ~* ^/storage/...` 把 `/storage/` 下的可执行后缀（php 家族 / cgi /
脚本类）请求一律拦成 404，且末尾的 `(/.*)?` 同时堵住 `PATH_INFO` 绕过（`/storage/x.php/y.jpg` 这种
借合法后缀骗过匹配、实际仍被当 php 跑的路径）。

**已有一批线上 vhost 要批量加固**时，仓库根带了幂等脚本 `tools/fix-nginx-storage-safe.sh`：

```bash
sh tools/fix-nginx-storage-safe.sh --dry-run              # 沙盘，只看会改哪些 .conf
sh tools/fix-nginx-storage-safe.sh                        # 实改（改前自动备份）
sh tools/fix-nginx-storage-safe.sh --dir /etc/nginx/conf.d  # 指定 vhost 目录
```

它扫描每个 `server {}` 块，把 legacy / 变体规则统一升级到当前标准，规则只插一次且排在第一条通用 PHP
location 之前。脚本**只改配置不重载**，改完手动 `nginx -t && nginx -s reload`。参数详见 [`SCRIPTS.md`](../SCRIPTS.md)。

> 上面只写 80 端口是为了聚焦骨架特有配置。`APP_URL` 既然是 https，生产还需要
> 443 + 证书 + 强制跳转——`certbot --nginx -d 你的域名` 一条命令即可补齐，细节略。

负载均衡 / 容器探活直接用健康检查路由：`GET /up` → 200。

## 8.5 缓存优化，以及一个必须知道的坑

```bash
php artisan optimize        # config / route / event 缓存一把梭
```

> ⚠️ **坑 #22（生产专属）**：`cache:clear` / `optimize:clear` 会清空 Redis 里的
> **JWT 黑名单**——所有"已登出 / 已续签作废"的 token 当场**复活**，直到自然过期。
> 部署脚本只跑 `php artisan optimize`（它重建配置/路由缓存，**不**清业务 cache）；
> 确实需要清 cache 时要意识到这个后果，安全敏感场景配合换 `JWT_SECRET` 强制全员重登。

### 顺手补全可观测三件套：慢 SQL 监听

异常上报（第 4 章）、操作日志（第 7 章）之外，moo-monitor-laravel（随 scaffold 3.9.0
自动带入）还内置了慢 SQL 监听——纯 env 开关，零代码：

```dotenv
MOO_MONITOR_SQL_SLOW_ENABLED=true
MOO_MONITOR_SQL_SLOW_THRESHOLD_MS=100     # 超过 100ms 的查询自动落盘
```

> 落盘目录是 `storage/moo-monitor/sql-slows`（自带 .gitignore，不入 git；和
> `storage/moo-monitor/runtimes` 一个套路）；storage 的写权限在 8.2 已经给过
> `www-data` 了，零额外配置。

落盘后在 `/scaffold` 的 S-Cloud 页可看缓冲状态。本仓库 `.env.example` 已带这两行。

接了 moo-scaffold-cloud 的话，`moo:cloud:push` 还能把 runtime / 慢 SQL 一并推上云，
接入只需两行 env：

```dotenv
MOO_MONITOR_CLOUD_ENABLED=true
MOO_MONITOR_CLOUD_TOKEN=…     # 云端「接入 Token」页生成，需同时具备 runtimes + slow_queries 能力
```

`enabled` 与 `schedule`（默认开）同真时，MonitorProvider 会自动挂**每分钟**的推送
调度——前提是宿主在跑调度器（8.6 的 `moo-schedule` 进程干的就是这件事）。其余可调项
（`MOO_MONITOR_CLOUD_URL` / batch / timeout 等）见
`vendor/charsen/moo-monitor-laravel/config/moo-monitor.php` 的 `cloud` 段注释。

## 8.6 进程守护（supervisor）

把下面内容存成 `/etc/supervisor/conf.d/moo.conf`，然后
`supervisorctl reread && supervisorctl update` 让它生效：

```ini
[program:moo-queue]
command=php /var/www/moo-engine-skeleton/engine/artisan queue:work redis --tries=3 --max-time=3600
; numprocs > 1 时必须配 process_name，否则 supervisord 直接拒绝加载该配置
process_name=%(program_name)s_%(process_num)02d
numprocs=2
autostart=true
autorestart=true
user=www-data

[program:moo-schedule]
command=php /var/www/moo-engine-skeleton/engine/artisan schedule:work
autostart=true
autorestart=true
user=www-data
```

> 改完 `.env` 或发新版后记得 `php artisan optimize && supervisorctl restart moo-queue:*`
> ——worker 是常驻进程，不重启拿不到新配置。第 7 章的**坑 #21**（`.env` 改了、
> worker 没重启、日志静默不落库，见 `07-安装-moo-system.md` 7.6 节）在生产同样成立。

## 8.8 数据库备份（backup.sh）

上线后第一件常被忘的运维事就是**备份**。仓库根的 `backup.sh` 做全库 mysqldump → bz2 压缩 →
落在 `engine/storage/app/db/` → 自动清过期备份，凭据默认从 `engine/.env` 读、环境变量可覆盖：

```bash
sh backup.sh                                  # 用 engine/.env 的 DB_* 配置
KEEP_DAYS=14 sh backup.sh                     # 旧备份保留 14 天（默认 7）
DB_HOST=10.0.0.10 DB_PORT=3306 sh backup.sh   # 备份远端库
```

默认 `--ignore-table` 跳过 `system_operation_logs`（操作日志表，体积大、不必入备份）。产物是
`<db>_YYYYMMDD_HHMMSS.tar.bz2`，还原用 `tar -xjf xxx.tar.bz2 && mysql -u root -p <db> < xxx.sql`。

**挂 cron 每天备份**（绝对路径 + 重定向日志，cron 的 `$PATH` 很短）：

```cron
0 3 * * * /opt/moo-engine-skeleton/backup.sh >> /var/log/moo-skeleton-backup.log 2>&1
```

> 备份是 8.9 的破坏性 schema 变更、以及 `DEPLOY-CHECKLIST.md` 里 Rollback 决策表的兜底前提——
> 数据层出问题时，代码可以 `pull.sh --tag` 秒回退，数据只能靠备份还原。

## 8.9 daily log 翻篇的 first-writer-wins 坑（cache.sh 治本）

Laravel 的 `daily` 日志通道每天 00:00 建一个新文件 `laravel-YYYY-MM-DD.log`（`auth-` / `dev-` 同理）。
**谁在 00:00 后第一个写日志，文件就归谁所有**。如果那一刻是 root 身份的 cron 或你 `sudo` 跑的某条
artisan 命令抢先建了文件 → 文件属主 `root:root` → 之后 PHP-FPM（`www-data`）想 append 就
`Permission denied` → 全站 500，且**只在每天凌晨复发一次**，极难现场抓。

仓库根的 `cache.sh` 把这个 race 治本，四层兜底：

1. `umask 002` —— 进程内建的文件都是 664（group 可写），不是默认 644；
2. `chmod 2775`（setgid）整个 `storage` 树 —— 新文件自动继承**目录的 group**（web group），
   而不是创建者的 primary group；
3. **预创建**今天 + 次日所有 daily channel 文件并立即 chown 给 web user —— 抢在 00:00 之前占好坑；
4. 末段 `reload php-fpm` —— 清掉 worker 内存里被 OPcache 缓存住的「这个文件不可写」失败状态
   （改对属主后不 reload，老 worker 仍会一直报 append failed）。

`cache.sh` 里的 daily channel 清单 `DAILY_LOG_BASENAMES="laravel auth dev"` 对应
`engine/config/logging.php` 的三个 `daily` 通道；host 新增 daily channel 时同步补一行。

**关键：cache.sh 要每天 cron 跑一次**——predcreate 只覆盖今天+明天，不每天跑则后天 00:00 又回到 race。
用 **web user**（不是 root）的 crontab：

```cron
55 23 * * * cd /opt/moo-engine-skeleton && sh cache.sh >/dev/null 2>&1
```

> 治本还有一步在 cache.sh 之外：queue worker / supervisor 的 `User=` 也要配成 web user（不要 root），
> 否则 worker 侧同样会抢先建 root-owned 日志。这条已写进 `DEPLOY-CHECKLIST.md` 的部署后 24h 守护项。
