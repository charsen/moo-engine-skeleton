# 第 8 章　部署上线（可选）

目标：把骨架部署到一台 Linux 服务器（nginx + php-fpm），并避开两个只在生产才会
暴露的坑：**雪花 ID / JWT 黑名单的共享缓存**、**清缓存导致已注销 token 复活**。

> 本章是通用部署流程的"骨架特化版"——只讲本项目特有的注意点，
> nginx/MySQL 的基础安装请参考各发行版文档。

---

## 8.1 服务器前提

- PHP 8.3（含 `pdo_mysql / mbstring / bcmath / redis` 扩展）+ Composer；
- MySQL 8 / MariaDB 10.6+，Redis；
- nginx + php-fpm；git（按 tag 拉代码）；certbot（HTTPS 证书，见 8.4）。

## 8.2 取码与依赖（composer 双轨制的"生产轨"）

> "双轨制"这个词第 2 章首次定义（`02-安装-moo-scaffold.md`）：**开发轨**用
> `composer.json`，moo-* 指向本地 path 仓库（`../../moo-scaffold`）；**生产轨**用
> `composer.production.json`，改为版本约束 + VCS。部署时把生产轨拷成 `composer.json`。

```bash
git clone <你的仓库> moo-engine-skeleton && cd moo-engine-skeleton/engine
git checkout <发布 tag>

# 切到生产 composer（moo-* 用版本约束 + VCS，而不是本地 path）
cp composer.production.json composer.json

# ⚠ 关键一步：仓库里提交的 composer.lock 是按"开发轨"解析的——
# 其中 moo-scaffold / moo-system / moo-monitor-laravel 的 dist 都指向本地 path（../../moo-scaffold），
# 生产服务器上没有这个目录，且 lock 的 content-hash 也和新 composer.json 对不上，
# 直接 composer install 必然失败。先删 lock，让生产轨重新解析：
rm composer.lock
composer update --no-dev --optimize-autoloader
```

> 想要可复现构建：第一次部署成功后，把生成的 lock 另存为
> `composer.production.lock` 进仓库，以后部署改为
> `cp composer.production.json composer.json && cp composer.production.lock composer.lock && composer install --no-dev`。

> 包的获取凭据：`composer.production.json` 里 moo-scaffold 和 moo-system **都**配置为
> Gitee 的 VCS 仓库（`git@gitee.com:charsen/…`），所以**现在两个包都需要在 Gitee
> 配只读 SSH 部署公钥**才拉得到。moo-scaffold 计划开源——发布到 Packagist
> 之后它才真正"无需凭据"；moo-system 是商业包，按授权配公钥。

### 目录权限（漏了这步，首个请求就 500）

php-fpm 以 `www-data` 运行，clone 出来的目录归属却是你的部署账号：

```bash
# Laravel 标配的两处可写目录
chown -R www-data:www-data storage bootstrap/cache
chmod -R ug+rwX storage bootstrap/cache

# 开了慢 SQL 监听（见 8.5）的话，落盘目录是【项目根】的 scaffold/——
# 不在 storage/ 下，也要给 www-data 写权限
chown -R www-data:www-data scaffold
```

## 8.3 生产 `.env` 要点

先创建 `.env` 并把数据库准备好（没有 `.env` 文件，下面的 `key:generate` 无处可写；
没建库，`migrate` 第一步就失败）：

```bash
cp .env.example .env
# 在 MySQL 里先建库（库名与 .env 的 DB_DATABASE 一致，示例用 moo_skeleton）：
#   CREATE DATABASE moo_skeleton CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

然后改这些键：

```dotenv
APP_ENV=production        # /scaffold 开发 UI 自动禁止写操作（机制见下方说明）
APP_DEBUG=false           # 必须关——异常堆栈里有路径和源码
APP_URL=https://你的域名   # 协议要与 nginx 实际监听一致，HTTPS 见 8.4

DB_DATABASE=moo_skeleton  # 连同 DB_USERNAME / DB_PASSWORD 换成生产值
DB_USERNAME=…
DB_PASSWORD=…

CACHE_STORE=redis         # ⚠ 必须：雪花 ID 跨 worker 共享、JWT 黑名单都存在 cache 里
QUEUE_CONNECTION=redis    # 操作日志/登录记录走队列，配 worker（见 8.6）
OPERATION_LOG=true
```

> ⚠ `.env.example` 里的 `REDIS_*` 几行默认是**注释掉的**（本地教程用不到）。
> 换 redis 时记得放开并填上真实连接——否则 worker 连不上 Redis，
> 8.7 冒烟项 7 会静默失败。

> **`APP_ENV=production` 到底"禁"了什么？** 写保护由 `EnforceScaffoldWritable`
> 中间件实现——production（或 readonly）模式下，`/scaffold` 的 db/designer、
> accounts、config、cloud/push 等写操作会被拒。而 **`moo:*` 命令与 APP_ENV 无关**：
> 它们一律 console-only（`ScaffoldProvider` 按 `runningInConsole()` 注册，HTTP
> 上下文任何环境都不注册），生产服务器的 CLI 里**照常可用**——8.5 在生产跑
> `moo:cloud:push` 靠的正是这一点。

首次部署生成密钥：

```bash
php artisan key:generate --force
php artisan jwt:secret --force     # ⚠ 只在首次！之后再换 = 所有已发 token 失效、全员重登
php artisan migrate --force
php artisan db:seed --force        # 按需：首批账号
php artisan vendor:publish --provider="Mooeen\Scaffold\ScaffoldProvider" --tag=public --force
```

> "换 `JWT_SECRET` = 全员重登"是同一行为的两面：日常**意外**换掉是事故；
> 而 8.5 坑 #22 的安全敏感场景里**主动**换，正是拿"全员重登"当手段。两处说法不矛盾。

## 8.4 nginx 站点

```nginx
server {
    listen 80;
    server_name 你的域名;
    root /var/www/moo-engine-skeleton/engine/public;
    index index.php;

    location / { try_files $uri $uri/ /index.php?$query_string; }

    location ~ \.php$ {
        fastcgi_pass unix:/run/php/php8.3-fpm.sock;
        fastcgi_index index.php;
        include fastcgi_params;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
    }
}
```

> 上面只写 80 端口是为了聚焦骨架特有配置。`APP_URL` 既然是 https，生产还需要
> 443 + 证书 + 强制跳转——`certbot --nginx -d 你的域名` 一条命令即可补齐，细节略。

负载均衡 / 容器探活直接用健康检查路由：`GET /up` → 200。

## 8.5 缓存优化，以及一个必须知道的坑

```bash
php artisan optimize        # config / route / event 缓存一把梭
```

> ⚠️ **坑 #22（生产专属）**：`cache:clear` / `optimize:clear` 会清空 Redis 里的
> **JWT 黑名单**——所有"已登出 / 已续签作废"的 token 当场**复活**，直到自然过期。
> 部署脚本只跑 `php artisan optimize`（它重建配置/路由缓存，**不**清业务 cache）；
> 确实需要清 cache 时要意识到这个后果，安全敏感场景配合换 `JWT_SECRET` 强制全员重登。

### 顺手补全可观测三件套：慢 SQL 监听

异常上报（第 4 章）、操作日志（第 7 章）之外，moo-monitor-laravel（随 scaffold 3.9.0
自动带入）还内置了慢 SQL 监听——纯 env 开关，零代码：

```dotenv
MOO_MONITOR_SQL_SLOW_ENABLED=true
MOO_MONITOR_SQL_SLOW_THRESHOLD_MS=100     # 超过 100ms 的查询自动落盘
```

> 落盘目录是 `storage/moo-monitor/sql-slows`（自带 .gitignore，不入 git；和
> `storage/moo-monitor/runtimes` 一个套路）；storage 的写权限在 8.2 已经给过
> `www-data` 了，零额外配置。

落盘后在 `/scaffold` 的 S-Cloud 页可看缓冲状态。本仓库 `.env.example` 已带这两行。

接了 moo-scaffold-cloud 的话，`moo:cloud:push` 还能把 runtime / 慢 SQL 一并推上云，
接入只需两行 env：

```dotenv
MOO_MONITOR_CLOUD_ENABLED=true
MOO_MONITOR_CLOUD_TOKEN=…     # 云端「接入 Token」页生成，需同时具备 runtimes + slow_queries 能力
```

`enabled` 与 `schedule`（默认开）同真时，MonitorProvider 会自动挂**每分钟**的推送
调度——前提是宿主在跑调度器（8.6 的 `moo-schedule` 进程干的就是这件事）。其余可调项
（`MOO_MONITOR_CLOUD_URL` / batch / timeout 等）见
`vendor/charsen/moo-monitor-laravel/config/moo-monitor.php` 的 `cloud` 段注释。

## 8.6 进程守护（supervisor）

把下面内容存成 `/etc/supervisor/conf.d/moo.conf`，然后
`supervisorctl reread && supervisorctl update` 让它生效：

```ini
[program:moo-queue]
command=php /var/www/moo-engine-skeleton/engine/artisan queue:work redis --tries=3 --max-time=3600
; numprocs > 1 时必须配 process_name，否则 supervisord 直接拒绝加载该配置
process_name=%(program_name)s_%(process_num)02d
numprocs=2
autostart=true
autorestart=true
user=www-data

[program:moo-schedule]
command=php /var/www/moo-engine-skeleton/engine/artisan schedule:work
autostart=true
autorestart=true
user=www-data
```

> 改完 `.env` 或发新版后记得 `php artisan optimize && supervisorctl restart moo-queue:*`
> ——worker 是常驻进程，不重启拿不到新配置。第 7 章的**坑 #21**（`.env` 改了、
> worker 没重启、日志静默不落库，见 `07-安装-moo-system.md` 7.6 节）在生产同样成立。

## 8.7 上线冒烟清单

| # | 验证 | 期望 |
|---|---|---|
| 1 | `GET /up` | 200 |
| 2 | 登录接口 | 200 + token |
| 3 | 无 token 调受保护接口 | 401（干净的 `{"message":...}`，无堆栈） |
| 4 | host `admin` 组接口带 token（如 `GET /api/admin/food`）响应头 | `X-RateLimit-Limit: 300`（注：moo-system 包路由如 `/api/admin/departments` 走 `moo-system` 组，鉴权排在限流前——无 token 先 401、不带此头，别拿它做本项冒烟） |
| 5 | 跨域请求响应头 | `Access-Control-Expose-Headers: Authorization` |
| 6 | `/scaffold` | 登录页可开、写操作被禁（production 模式） |
| 7 | 操作日志 | `system_operation_logs` 有新记录（说明 queue worker 活着） |
