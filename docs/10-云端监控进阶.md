# 第 10 章　云端监控进阶

目标：深入使用 moo-scaffold-cloud（`https://sc.mooeen.com`）的聚合告警、AI 辅助处理、
多项目管理等进阶功能，并掌握从 moo-scaffold ≤3.8 迁移旧监控数据的方法。

> **前置要求**：第 1.7 节已接入 `moo-monitor-laravel`，本地异常采集能正常工作，
> 且已拿到 cloud token 并至少成功推送过一次。本章聚焦**云端能力**，
> 不涉及本地采集配置（那些在第 1.7 节已完成）。

## 10.1 云端项目管理

### 注册与创建项目

访问 `https://sc.mooeen.com`，用 GitHub / Gitee / 邮箱注册账号。
免费档支持 **≤3 个项目**（每个项目独立 token，互不干扰）。

登录后点击「新建项目」，填写：

- **项目名称**：如 `moo-engine-skeleton`（仅用于云端显示，与本地项目目录名无关）
- **启用能力**：勾选 `runtimes`（运行时异常）和 `slow_queries`（慢 SQL）

创建后自动生成 **接入 token**（以 `moo_` 前缀开头），
复制到本地 `.env` 的 `MOO_MONITOR_CLOUD_TOKEN`。

### 多项目切换

云端左上角项目下拉菜单可快速切换。每个项目独立统计：

- 运行时异常总数 / open / resolved / deleted
- 慢 SQL 总数 / open / resolved / deleted
- 最近 7 天趋势图

### Token 权限

一个 token 绑定一个项目，推送时云端按 token 自动路由到对应项目。
**Token 泄露风险**：他人拿到你的 token 只能往你的项目推数据（无法读取或删除）——
影响有限，但仍建议定期轮换。云端「设置 → 接入 Token」页可重新生成。

## 10.2 运行时异常：聚合与处理流程

### 10.2.1 按 hash 聚合

相同异常（类型 + 消息 + 文件 + 行号生成的 hash）自动聚合为一条记录，
累计 `count`（总出现次数）和 `daily.count`（今日出现次数）。
云端列表默认按 `last_seen` 倒序，最近出现的排最前。

点击某条记录进入详情页，能看到：

- 异常类型、消息、触发位置（文件 `:` 行号，可点击跳转源码——需配置代码托管地址）
- 完整调用栈（`trace.full`）和简化栈（`trace.simplified`，过滤框架内部）
- 首次 / 最近出现时间、累计次数、今日次数
- 请求信息（method / url / ip / user_id / user_name / guard）
- 上下文（env / project / php / laravel / occurred_at）

### 10.2.2 处理状态机

每条记录有三种状态：

- **open**（待处理）：新推送上来的默认状态
- **resolved**（已解决）：标记后不再在 open 列表显示；若同一异常再次出现，
  云端自动重开为 open（`count` 累加，`resolved_at` 清空）
- **deleted**（已删除）：彻底删除（仅云端删除，本地缓冲不受影响）

**操作入口**：详情页右上角「标记已解决」/「删除」按钮，或列表批量操作。

**处理备注**：标记 resolved 时可填写 `resolved_note`（如"修复提交 abc123"），
记录在 `resolved_by`（操作人账号）和 `resolved_at`（操作时间）。

### 10.2.3 告警规则（付费功能）

免费档无告警。付费档可配置：

- **频次告警**：同一异常 1 小时内出现 ≥N 次 → 推送企业微信 / 钉钉 / 邮件
- **新异常告警**：首次出现的 hash → 立即通知
- **静默期**：同一异常告警后 X 小时内不再重复推送

## 10.3 慢 SQL：阈值与优化线索

### 10.3.1 本地阈值配置

`.env` 中 `MOO_MONITOR_SQL_SLOW_THRESHOLD_MS`（默认 100ms）决定哪些 SQL 被记录。
建议：

- **开发环境**：100ms（足够敏感，及时发现 N+1）
- **生产环境**：500ms 或更高（避免高频查询刷屏，聚焦真正的瓶颈）

记录字段包括：

- `sql`（参数化 SQL，`?` 占位符）、`bindings`（实际参数值，敏感字段已脱敏）
- `time`（执行耗时，毫秒）
- `location`（触发位置：文件 + 行号）
- `request`（method / url / ip / user）

### 10.3.2 云端分析

慢 SQL 列表按 `time` 倒序排列，点开详情能看到：

- SQL 全文（带实际参数值，便于本地 `EXPLAIN` 复现）
- 触发位置（快速定位哪个控制器 / 方法在跑这条查询）
- 累计出现次数（高频慢查询 = 优先优化对象）

**优化线索**：

- 缺索引 → 对照 `EXPLAIN` 结果，加索引后重跑
- N+1 问题 → 同一个查询短时间内重复出现多次，改用 `with()` 预加载
- 全表扫描 → `WHERE` 条件字段没索引，或 `LIKE '%xxx%'` 前缀通配

## 10.4 AI 辅助处理：MCP 接入（🌟 全教程独有亮点）

### 10.4.1 什么是 MCP

**MCP（Model Context Protocol）** 是 Anthropic 定义的标准协议，
让 AI 工具（Claude Code / Codex 等）能通过统一接口读取外部数据源。
`moo:cloud:mcp` 把云端的 runtime 错误和待办暴露为 MCP server，
AI 可以：

1. **拉取**云端 open 状态的异常列表（按项目 + 最近 7 天）
2. **查看**某条异常的详情（调用栈、请求信息、上下文）
3. **回写**处理状态（标记 resolved + 填写备注）

### 10.4.2 接入 Claude Code

在 `engine/` 目录执行（仓库根目录没有 `artisan`，从根目录跑 bare `php artisan` 会找不到命令、MCP server 会静默起不来）：

```bash
claude mcp add moo-cloud -- php artisan moo:cloud:mcp
```

> Claude Code 默认（local scope）把这条命令写入 `~/.claude.json` 中对应项目路径下的 `mcpServers`（不是 `~/.claude/settings.json`，那里没有 `mcpServers` 字段），
> 以后每次启动 Claude Code 会话时自动连接。

MCP 复用 `.env` 中的 `MOO_MONITOR_CLOUD_URL` 和 `MOO_MONITOR_CLOUD_TOKEN`，
不需要额外配置。

### 10.4.3 工作流演示

典型对话：

```
你：从云端拉最近的 runtime 错误，挑一个看看怎么修。

Claude Code（调用 MCP）：
拉到 3 条 open 异常，最高频的是 `RuntimeException` in `FoodController.php:42`，
出现 12 次，消息是「库存不足」。我去读一下那个文件……

[读取 app/Admin/Controllers/Food/FoodController.php]

发现问题：第 42 行直接抛异常，没有提前检查库存。建议改成：
if ($food->stock < $quantity) {
    throw new BaseException('库存不足', 522);
}
并在前面加 DB 事务包裹。要我改吗？

你：改吧，改完标记已解决。

Claude Code：
[修改文件、运行测试]
修改完成，测试通过。现在标记云端异常为 resolved……
[调用 MCP 回写]
已标记，备注：「修复提交 abc123 - 增加库存预检查」。
```

**价值**：AI 不需要你手动复制粘贴错误信息，直接从云端拉真实生产数据、
看完整上下文、改代码、回写处理状态——**闭环自动化**。

### 10.4.4 MCP 可用工具

`moo:cloud:mcp` 暴露的 MCP tools：

- `list_open_runtimes`：列出云端「待处理」的 runtime 错误（默认 open + in_progress，最近优先）。可选参数：`limit`（1–50，默认 20）/ `status`（`open` / `in_progress` / `resolved`）
- `get_runtime`：读取某条异常的详情（参数：`hash`）
- `resolve_runtime`：标记已解决（参数：`hash` / `note`）
- `list_open_todos`：列出云端「可处理」的待办（测试 / 产品提报的 bug 单，默认 open + in_progress，最新优先）。可选参数：`limit`（1–50）/ `status`（`open` / `in_progress` / `done`）
- `get_todo`：读取某条待办的完整上下文（参数：`id`）
- `update_todo_status`：回写待办状态——认领（`in_progress`）或完成（`done`）（参数：`id` / `status` / `note`）

## 10.5 从 moo-scaffold ≤3.8 迁移

### 10.5.1 背景

moo-scaffold 3.9.0 把监控拆给了 `moo-monitor-laravel` 包，旧版项目升级后：

- 旧数据在 `scaffold/runtimes/` 和 `scaffold/sql-slows/`
- 新数据在 `storage/moo-monitor/runtimes/` 和 `storage/moo-monitor/sql-slows/`

两套目录并存，推送命令只认新目录。

### 10.5.2 迁移步骤

```bash
# 1. 升级包
composer update charsen/moo-scaffold

# 2. 执行迁移（自动把 scaffold/* 移动到 storage/moo-monitor/*，并删除清空后的旧目录）
php artisan moo:monitor:migrate

# 3. 预演推送（确认待推数量）
php artisan moo:cloud:push --dry-run

# 4. 真实推送
php artisan moo:cloud:push
```

迁移命令会：

- 把 `scaffold/runtimes/{open,resolved,deleted}/` 下所有 `.yaml` 移动到
  `storage/moo-monitor/runtimes/` 对应目录
- 把 `scaffold/sql-slows/` 同理迁移
- 迁移后旧 `scaffold/runtimes`、`scaffold/sql-slows` 等已清空的目录会被自动删除（连同残留的 `.gitkeep` / `.gitignore`），命令会打印「旧目录空壳已清理」
- 更新 `storage/moo-monitor/cloud-sync.json` 的游标，确保迁移后的旧数据也能被推送

### 10.5.3 环境变量改名

旧项目 `.env` 需要改前缀：

| 旧变量 | 新变量 |
|---|---|
| `SCAFFOLD_RUNTIME_ENABLED` | `MOO_MONITOR_RUNTIME_ENABLED` |
| `SCAFFOLD_RUNTIME_MAX_OPEN` | `MOO_MONITOR_RUNTIME_MAX_OPEN` |
| `SCAFFOLD_SQL_SLOW_ENABLED` | `MOO_MONITOR_SQL_SLOW_ENABLED` |
| `SCAFFOLD_SQL_SLOW_THRESHOLD_MS` | `MOO_MONITOR_SQL_SLOW_THRESHOLD_MS` |
| `SCAFFOLD_CLOUD_ENABLED` | `MOO_MONITOR_CLOUD_ENABLED` |
| `SCAFFOLD_CLOUD_TOKEN` | `MOO_MONITOR_CLOUD_TOKEN` |
| `SCAFFOLD_CLOUD_URL` | `MOO_MONITOR_CLOUD_URL` |

改完后 `php artisan config:clear`。

## 10.6 生产环境配置

### 10.6.1 自动推送

在 `.env` 中同时开启：

```dotenv
MOO_MONITOR_CLOUD_ENABLED=true
MOO_MONITOR_CLOUD_SCHEDULE=true
```

MonitorProvider 会自动注册 **每分钟推送任务**（Laravel scheduler）。
服务器 crontab 需要跑 `schedule:run`：

```cron
* * * * * cd /path/to/project/engine && php artisan schedule:run >> /dev/null 2>&1
```

没有 scheduler 的环境，可以自己写 cron：

```cron
*/5 * * * * cd /path/to/project/engine && php artisan moo:cloud:push >> /dev/null 2>&1
```

### 10.6.2 本地缓冲保留期

推送成功后，本地 `.yaml` 文件默认保留 **7 天**（`MOO_MONITOR_CLOUD_LOCAL_RETENTION_DAYS=7`），
之后自动清理。生产环境磁盘紧张时可改成 3 天或 1 天；
开发环境建议保留 14 天，便于对照本地文件调试。

### 10.6.3 私有部署

`MOO_MONITOR_CLOUD_URL` 默认是 `https://sc.mooeen.com`（官方 SaaS）。
企业私有部署时改成自己的域名即可，推送协议与官方版完全兼容。

## 10.7 常见问题

**排查任何「数据上不了云」的问题，第一步先跑一键自检 `moo:cloud:test`**——它把
「配置 → 连通 → 鉴权 → 推送」四步逐个走一遍并分别反馈，直接告诉你断在哪：

```bash
php artisan moo:cloud:test
```

- 卡在 **① 配置检查** → token / URL 没配（`.env` 的 `MOO_MONITOR_CLOUD_URL` / `_TOKEN`）；
- 卡在 **② 心跳失败** → 云端不可达或 token 无效（见下方「401」）；内网自签证书可设 `MOO_MONITOR_CLOUD_VERIFY=false`；
- **③④ 推送失败** → 多是本地数据格式与云端 API 版本不匹配（见下方「422」）。

比直接 `moo:cloud:push` 推完一片沉默好定位得多；下面各条是缩小范围后的具体修法。

### 推送失败：401 Unauthorized

原因：token 无效或已过期。检查：

1. `.env` 中 `MOO_MONITOR_CLOUD_TOKEN` 是否完整复制、`moo_` 前缀齐全、无空格
2. 云端项目是否被删除
3. 云端账号是否欠费停用（付费档）

### 推送失败：422 Validation Error

原因：本地数据格式不符合云端 API 要求（通常是版本不匹配）。解决：

```bash
composer update charsen/moo-monitor-laravel
php artisan config:clear
php artisan moo:cloud:push --all   # 忽略游标，全量重推（云端按 hash 幂等）
```

### MCP 连接失败

检查：

1. `php artisan moo:cloud:mcp` 能否单独跑通（不报错即正常）
2. `~/.claude.json` 中对应项目路径下 `mcpServers.moo-cloud` 条目是否存在（local scope 默认写在这里，不在 `~/.claude/settings.json`）
3. `.env` 中 `MOO_MONITOR_CLOUD_URL` / `_TOKEN` 是否配置

### 异常没有被记录

检查：

1. `.env` 中 `MOO_MONITOR_RUNTIME_ENABLED=true`
2. 异常是否在 `bootstrap/app.php` 的 `dontReport` 列表里
3. `php artisan config:clear` 清理缓存后重试
4. 查看 `storage/logs/laravel.log` 是否有 MonitorProvider 报错
