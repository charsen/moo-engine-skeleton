# 第 11 章　操作人身份契约与扩展包接入

目标：让 host 和所有 Moo Laravel 扩展包对“当前操作人”使用同一套语义，避免每个包复制 `HasOperator`、直接猜测 guard，或用 `0` 伪造人员身份。

## 11.1 最终契约

`moo-scaffold 2.1.3` 起提供公共契约：

```php
use Mooeen\Scaffold\Contracts\OperatorResolver;

$operatorId = app(OperatorResolver::class)->id();
```

返回值只有三种：真实的 `int` ID、真实的 `string` ID，或者未登录时的 `null`。人员 ID 不使用 `0` 或 `'0'` 作为“系统动作”哨兵。

scaffold 默认通过 `bindIf()` 绑 `Support\GuardOperatorResolver`（裸 `auth()->id()`）。host 若想统一走自己的取值口径，在自己的 Provider 里 `bind()` 覆盖即可——`bindIf` 的语义保证 host 绑定稳定生效。

## 11.2 host 覆盖示范（App\Moo\<包> 接入约定）

本骨架**已落地一份覆盖**作为示范：把操作人身份收敛到全站统一的 `getUserId()` 口径，而不是让各处直接 `auth()->id()`（未来包默认实现与 host helper 逻辑漂移就会踩坑）。

落点遵循 **`App\Moo\<包>` 约定**：host 对某个 Moo 生态包的本地接入/覆盖，统一收在 `app/Moo/<包名>/` 下自己的命名空间里，而不是去改 `vendor/`。这样包能独立升级、host 的定制不丢。

`app/Moo/Scaffold/GetUserIdOperatorResolver.php` —— host 侧实现：

```php
namespace App\Moo\Scaffold;

use Mooeen\Scaffold\Contracts\OperatorResolver;

class GetUserIdOperatorResolver implements OperatorResolver
{
    public function id(): int|string|null
    {
        // getUserId() 读 config('auth.defaults.guard')（JWTAssignGuard 中间件按路由设默认守卫；游客返 null）
        return getUserId();
    }
}
```

`app/Moo/Scaffold/ScaffoldServiceProvider.php` —— host 侧 Provider，在 `register()` 里 `bind` 覆盖：

```php
namespace App\Moo\Scaffold;

use Illuminate\Support\ServiceProvider;
use Mooeen\Scaffold\Contracts\OperatorResolver;

class ScaffoldServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        $this->app->bind(OperatorResolver::class, GetUserIdOperatorResolver::class);
    }
}
```

该 Provider 注册在 `bootstrap/providers.php`。覆盖一处，全生态所有 `HasOperator`（`creator_id` / `updater_id` 自动填充）随之统一口径。绑定应只出现在 host 的**一个** Provider 中；扩展包不得覆盖 host 绑定，也不得重新读取自己的 guard。

## 11.3 模型统一使用共享 HasOperator

需要自动填充 `creator_id` / `updater_id` 的模型直接引用共享 Trait：

```php
use Mooeen\Scaffold\Concerns\HasOperator;

class Food extends Model
{
    use HasOperator;
}
```

不要在 `app/Models/Traits` 或扩展包中复制 `HasOperator.php`。新版 `moo:model` 生成器会直接导入 scaffold 的共享 Trait。

共享 Trait 只对 `$fillable` 中存在的审计字段赋值。未登录时写入 `null`，所以数据库中的人员审计列应允许 nullable。

## 11.4 各消费者如何处理 null

公共契约只回答“当前是谁”，业务边界自行决定无身份策略：

| 场景 | 无有效身份时 |
|---|---|
| 普通模型审计列 | 保持 `null` |
| 附件直传签发 | 拒绝 `null`、`''`、`0`、`'0'` |
| 行为轨迹 | 跳过记录，不落伪身份 |
| radar 人工处理 | 拒绝闭环操作 |
| 队列任务 | 派发前快照 ID，worker 不重新读取登录态 |

如果业务确实需要表示“系统动作”，请增加动作来源字段或系统主体模型，不要占用人员外键的 `0`。

## 11.5 版本与验证

生产环境最低稳定版本：

```json
{
  "charsen/moo-scaffold": "^2.1.3",
  "charsen/moo-system": "^1.6.15"
}
```

本地开发可继续使用 path/VCS 开发分支；生产配置应声明 scaffold 的 VCS 仓库并从稳定 tag 安装（同步到 Packagist 后才可删除该仓库声明）：

```json
{
  "repositories": {
    "scaffold": {
      "type": "vcs",
      "url": "https://gitee.com/charsen/moo-scaffold.git"
    }
  }
}
```

验证命令：

```bash
composer validate
composer install --no-dev --classmap-authoritative
php artisan test --filter=OperatorResolverTest
rg -n "App\\\\Models\\\\Traits\\\\HasOperator|trait HasOperator|auth\\(\\)->id" app packages
```

最后一个搜索允许业务控制器按当前 guard 做资源归属判断，但不应再发现本地 `trait HasOperator` 或扩展包自建的“当前操作人”契约。

完成本章后，host、生成模型和扩展包使用的是同一身份来源；游客是 `null`，人员 ID 是真实 ID，队列拿到的是派发时快照。
